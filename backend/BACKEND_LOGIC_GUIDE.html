auto-K8S Backend Guide

Backend Logic, Packages, and Imports

This document explains the backend part of your auto-K8S project: what each file does, how the Django API works, how the database is structured, and how the backend launches an Ansible playbook through WSL to install Kubernetes on target machines.

Open this file in a browser, then use Ctrl+P and choose Save as PDF if you want a PDF version.

1. Backend Goal

The backend has one main job: receive data from the frontend, store machine and credential information in PostgreSQL, and trigger an Ansible playbook that installs Kubernetes on selected hosts.

What the backend stores

  • Credentials used to connect by SSH
  • Hosts to manage
  • Playbook templates
  • Installation history and target results

What the backend exposes

  • REST API endpoints for hosts
  • Read endpoints for credentials and templates
  • An endpoint to launch an installation
  • An endpoint to read installation history

What the backend executes

  • Dynamic inventory generation
  • WSL path conversion
  • ansible-playbook command execution
  • Capture of stdout and stderr for debugging

5. Database Structure

The database is intentionally simple. It stores only the information needed to launch Kubernetes installation.

Model Role Main fields
Credential Stores how to connect to a host name, auth_type, username, private_key, password, become_password
Host Represents one target machine name, hostname, ip_address, ssh_port, enabled, credential
InstallationTemplate Represents a playbook choice name, playbook_path, description
Installation Represents one launch request template, status, started_at, finished_at
InstallationTarget Stores the result of one host inside one installation installation, host, status, output

Why this schema is good here

  • It separates connection data from host data. Several hosts can reuse the same credential.
  • It separates the playbook template from the execution history.
  • It keeps one global installation record and one per-target result record.
  • It is minimal and enough for launching installations and debugging failures.

6. Serializer Logic

Serializers are the bridge between Python objects and JSON.

CredentialSerializer

Validates credential data before it is saved.

  • If auth_type = ssh_key, then private_key must be present.
  • If auth_type = password, then password must be present.

HostSerializer

Exposes host fields plus a read-only credential_name field for easier display in the frontend.

InstallationTemplateSerializer

Returns playbook template information.

InstallationTargetSerializer

Returns target execution information plus a read-only host_name.

InstallationSerializer

Returns installation data plus nested targets and a read-only template_name.

Without serializers, the backend would need much more manual JSON and validation code.

7. API Endpoints

Method Endpoint Purpose
GET, POST /api/hosts/ List all hosts or create a new host
GET, PUT, PATCH, DELETE /api/hosts/<id>/ Read, replace, partially update, or delete one host
GET /api/credentials/ List credentials
GET /api/templates/ List playbook templates
GET /api/installations/ List execution history
GET /api/installations/<id>/ Read one installation and its targets
POST /api/installations/launch/ Launch the Ansible playbook for selected hosts

8. View Logic

hosts_list_create

  • GET: reads all Host records and returns them as JSON.
  • POST: validates request data with HostSerializer and saves a new host.

host_detail

  • GET: returns one host.
  • PUT: replaces the host with full new data.
  • PATCH: updates only some fields.
  • DELETE: removes the host.

credentials_list and templates_list

These are simple read-only endpoints used by the frontend select boxes.

installations_list and installation_detail

These allow you to inspect past executions and debug output later.

launch_installation

This is the core backend function. It does all of the following:

  1. Reads template_id and host_ids from the request.
  2. Validates that the template and hosts exist.
  3. Creates one Installation row with status pending, then running.
  4. Creates one InstallationTarget row for each selected host.
  5. Builds a temporary dynamic inventory file.
  6. Locates the Ansible repository on disk.
  7. Calls run_playbook().
  8. Collects stdout and stderr.
  9. Marks the installation and targets as success or failed.
  10. Returns a detailed JSON response to the frontend.

9. Service Layer Logic

The backend keeps Ansible-related logic in services.py. This is good because the view stays smaller and more readable.

build_inventory_file(hosts)

This function creates a temporary YAML inventory for the selected hosts.

all:
  children:
    selected_hosts:
      hosts:
        target2:
          ansible_host: 192.168.100.133
          ansible_port: 22
          ansible_user: "target2"
          ansible_password: "0000"
          ansible_become_password: "0000"

Why this was important:

  • It avoids depending on a static inventory file.
  • It allows the web app to select hosts dynamically.
  • Using YAML avoids the earlier bug where 0000 became integer 0.

windows_to_wsl_path(path)

This converts a Windows path like C:\Users\ASUS\... into a WSL path like /mnt/c/Users/ASUS/....

It is required because Django runs on Windows while Ansible is executed inside Ubuntu WSL.

run_playbook(repo_root, playbook_relative_path, inventory_path)

This function:

  • Checks that the playbook, ansible.cfg, and roles folder exist.
  • Converts those paths to WSL format.
  • Builds the shell command.
  • Runs ansible-playbook using subprocess.run().
  • Returns the command result so the API can display the output.
["wsl", "-d", "Ubuntu", "sh", "-lc", shell_command]

This is the critical bridge between your Django backend and the Ansible automation engine.

10. Full Backend Execution Flow

  1. User selects hosts in the React frontend.
  2. User selects the playbook template.
  3. Frontend sends a POST request to /api/installations/launch/.
  4. Django validates the payload.
  5. Django reads selected hosts and their credentials from PostgreSQL.
  6. Django generates a temporary inventory file.
  7. Django locates the Ansible repository.
  8. Django runs Ansible inside Ubuntu WSL.
  9. Ansible connects to the targets and runs the playbook roles.
  10. Django captures the command output and stores the result.
  11. Frontend displays success, failure, stdout, and stderr.

12. What Is Still Important for Portability

  • The database settings are already portable because they come from environment variables.
  • The backend still assumes Ansible runs inside WSL Ubuntu on Windows.
  • The backend still expects a local cloned Ansible repository with ansible.cfg, roles, and playbooks.
  • If you want full portability later, make the Ansible repo path configurable from an environment variable like ANSIBLE_REPO_ROOT.
  • You can also validate credentials before launching to fail earlier when a host has no valid SSH user or password.

14. Summary

Your backend is now a simple but solid orchestration layer: PostgreSQL stores the data, Django REST API exposes the endpoints, serializers validate the input, services prepare the inventory and launch Ansible, and the installation result is stored and returned to the frontend.

For a first version, the structure is coherent: models are minimal, views are understandable, and the Ansible execution path is separated from the CRUD logic.
