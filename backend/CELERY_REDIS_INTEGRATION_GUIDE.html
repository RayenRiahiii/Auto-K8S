auto-K8S Backend Guide

Celery + Redis Integration

This document explains exactly how Redis and Celery were integrated into the current auto-K8S backend, what each file does, how the configuration works, and how a playbook execution now moves from the Django API to a background worker.

Open this file in a browser. If you want a PDF, use Ctrl+P then choose Save as PDF.

1. Why We Added Celery and Redis

Before Celery, the Django view launched Ansible directly inside the HTTP request. That meant the browser request stayed open until ansible-playbook finished.

That was a problem because Kubernetes installation can take several minutes. The browser waited too long, the backend request was blocked, and the UI could not reflect a proper background job model.

Celery role

Celery is the background task executor. It receives jobs from Django and runs them outside the HTTP request.

Redis role

Redis is the message broker and result backend. Celery sends tasks into Redis and the worker consumes them.

Frontend role

The frontend no longer waits for the full playbook response. It polls the installation status instead.

2. High-Level Execution Flow

  1. User selects a playbook template and one or more hosts in the frontend.
  2. Frontend sends POST /api/installations/launch/ to Django.
  3. Django validates the data and creates an Installation row plus InstallationTarget rows.
  4. Django queues a Celery task with run_installation_task.delay(installation.id).
  5. Django immediately returns a 202 Accepted response with the installation id.
  6. Celery worker receives the task through Redis.
  7. Celery calls the installation execution logic in the service layer.
  8. The service layer builds inventory, runs Ansible, stores output, and updates database statuses.
  9. The frontend polls /api/installations/<id>/ until the status becomes success or failed.

3. Files Added or Changed

File Purpose
backend/backend/celery.py Creates the Celery application object for the Django project.
backend/backend/__init__.py Exposes the Celery app so Django and Celery can discover it in the standard way.
backend/backend/settings.py Defines Redis broker/backend configuration for Celery.

4. Celery App Configuration

backend/backend/celery.py

import os

from celery import Celery

os.environ.setdefault("DJANGO_SETTINGS_MODULE", "backend.settings")

app = Celery("backend")
app.config_from_object("django.conf:settings", namespace="CELERY")
app.autodiscover_tasks()

What this does:

  • os.environ.setdefault(...) tells Celery which Django settings module to use.
  • Celery("backend") creates the Celery app for this Django project.
  • config_from_object(..., namespace="CELERY") loads only settings that start with CELERY_.
  • autodiscover_tasks() tells Celery to look for tasks.py files inside installed Django apps.

5. Django Project Hook

backend/backend/__init__.py

from .celery import app as celery_app

__all__ = ("celery_app",)

This file makes the Celery app available as part of the Django project package. It is the standard pattern used so that Celery sees the project app correctly.

6. Redis Configuration in Django Settings

backend/backend/settings.py

CELERY_BROKER_URL = "redis://localhost:6379/0"
CELERY_RESULT_BACKEND = "redis://localhost:6379/0"
CELERY_ACCEPT_CONTENT = ["json"]
CELERY_TASK_SERIALIZER = "json"
CELERY_RESULT_SERIALIZER = "json"
CELERY_TIMEZONE = "UTC"

Meaning of each setting:

  • CELERY_BROKER_URL: where Celery sends tasks. Here, Redis on local port 6379.
  • CELERY_RESULT_BACKEND: where task results are stored. Also Redis.
  • CELERY_ACCEPT_CONTENT: accepted payload format.
  • CELERY_TASK_SERIALIZER: format used to serialize tasks into Redis.
  • CELERY_RESULT_SERIALIZER: format used to serialize stored results.
  • CELERY_TIMEZONE: timezone used by Celery.

In the current local setup, Redis is expected on localhost:6379. If Redis is not running there, Celery will fail to queue or consume tasks.

7. Python Dependencies

backend/requirements.txt

celery==5.5.3
redis==6.4.0

Why they are needed:

  • celery: the task queue system.
  • redis: Python client used by Celery to talk to Redis.

django-redis is not required for Celery broker usage. It is only needed if Django cache/session should use Redis too.

8. The Correct Separation: Views, Tasks, Services

Views Layer

views.py should validate requests, create database rows, and queue the background task. It should not run long Ansible execution directly anymore.

Tasks Layer

tasks.py should stay thin. Its job is to expose Celery tasks and forward the real work to the service layer.

Services Layer

services.py contains the heavy business logic: generate inventory, run Ansible through WSL, measure timings, update the installation state, and store output.

9. Current Task Wrapper

backend/auto_K8S/tasks.py

from celery import shared_task

from .services import execute_installation


@shared_task
def run_installation_task(installation_id):
    return execute_installation(installation_id)


@shared_task
def debug_task():
    print("debug task executed")
    return "ok"

run_installation_task is now the correct Celery entry point. It receives only installation_id, not whole model objects.

This is important because Celery tasks should pass small serializable values, then query the database again inside the worker.

10. Current Service Execution Logic

The main background job logic lives in services.py inside execute_installation(installation_id).

It performs these steps:

  1. Load the Installation and related InstallationTarget rows from the database.
  2. Mark installation and targets as running.
  3. Build a temporary dynamic Ansible inventory file.
  4. Locate the Ansible repository root.
  5. Run ansible-playbook inside Ubuntu WSL.
  6. Measure execution timings.
  7. Parse slow tasks and bottleneck hints.
  8. Store combined stdout/stderr into each target output.
  9. Update the installation status to success or failed.

This separation is important because the same service can be tested, reused, and reasoned about independently of Celery.

11. View Refactor for Background Launch

The launch API was changed so that it no longer blocks until Ansible completes.

Before

  • Django view ran the full playbook itself.
  • HTTP response returned only when Ansible finished.

After

  • Django creates installation rows.
  • Django calls run_installation_task.delay(installation.id).
  • Django immediately returns 202 Accepted.

12. Redis Startup on Windows with Docker

The local integration uses Docker Desktop to run Redis.

docker run --name django-redis -d -p 6379:6379 --rm redis

Useful verification commands:

docker ps
docker logs django-redis
docker exec -it django-redis redis-cli ping

Expected Redis ping response:

PONG

13. Starting the Worker on Windows

The worker should be started from the backend directory, and on Windows the recommended pool is solo. 

cd C:\Users\ASUS\Desktop\auto-K8S\backend
.venv\Scripts\celery.exe -A backend worker --loglevel=info --pool=solo

Why --pool=solo matters:

  • Windows often has issues with Celery multiprocessing pools.
  • solo is simpler and more reliable for development on Windows.

14. What Healthy Celery Logs Look Like

transport: redis://localhost:6379/0
results: redis://localhost:6379/0
[tasks]
  . auto_K8S.tasks.debug_task
  . auto_K8S.tasks.run_installation_task

celery@DESKTOP-... ready
Task auto_K8S.tasks.run_installation_task[...] received
Task auto_K8S.tasks.run_installation_task[...] succeeded in ...s

If you see received and then succeeded, that means Celery and Redis are working together correctly.

Be careful: a Celery task can succeed as a Python function but still return an installation result where Ansible failed. That means the queue worked, but the infrastructure action did not.

15. Frontend Behavior After Celery

The frontend had to change because the backend no longer returns the final playbook output immediately.

  • The launch endpoint now returns only a queued job response.
  • The frontend stores installation_id.
  • The frontend polls /api/installations/<id>/.
  • While polling, the UI shows a percentage, elapsed time, and progression state.
  • When the status becomes success or failed, the frontend reads the stored output from the API.

16. Common Problems We Hit and Why

Problem Cause Fix
Celery app not found Wrong worker command like -A tasks Use celery -A backend worker ...
Redis connection refused Redis not running on localhost:6379 Start Redis container
Windows pool permission errors Default pool model on Windows Use --pool=solo
Circular import between views and tasks Task imported helper from view Move shared logic into services.py
run_playbook shown as a Celery task It was incorrectly decorated or registered Keep it as a normal service function only
Frontend stayed at pending Frontend expected synchronous final response Add polling of installation status

17. Current State of the Integration

  • Celery app configuration exists and is valid.
  • Redis is used as broker and result backend.
  • Installation launch is asynchronous.
  • Worker execution logic is separated from HTTP views.
  • Frontend polls background installation status.
  • Windows worker startup requires --pool=solo.

In short: Redis and Celery were integrated successfully as the queue/execution layer for long Ansible playbook runs.

18. Next Improvements

  • Persist rich metadata like command, timings, inventory, stdout, and stderr directly in the installation model.
  • Move Celery broker settings to environment variables instead of hardcoding localhost.
  • Add installation history in the frontend.
  • Add cancellation or retry support later if needed.
  • Add a dedicated async model if SSH verification should also become a background job.
